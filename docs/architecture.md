# Project Soltra: Systems Architecture & Blueprint

## Project Overview
**Project Soltra** is a commercial-grade, distributed edge-computing solar tracking array prototype. It is a decentralized, AI-driven, weather-predicting, computer-vision solar tracking system featuring a zero-collision radio network, a cloud telemetry backend, a voice-interactive AI operator HUD, and a highly optimized power budget.

This document serves as the master context and architecture blueprint for the IDE and external agents to understand the system state and current engineering focus.

---

## 1. Core Processing Array (The Brains)

* **The Master Hub (Central Command):** `Heltec WiFi LoRa 32 (V3)`
  * **Role:** The central conductor, radio master, cloud gateway, and physical motor driver.
  * **Firmware:** `soltra_master_hub.ino` — Phase 5 active firmware.
  * **OS/Architecture:** Dual-Core FreeRTOS, permanently pinned:
    * *Core 0 (cloudTask):* Wi-Fi provisioning (WiFiManager captive portal), HiveMQ TLS cloud link (port 8883), JSON telemetry publish every 5s, subscribe to `helios/control/manual` and route commands via ESP-NOW to motor node.
    * *Core 1 (radioTask):* Strictly dedicated to catching ESP-NOW radio packets from sensor corners, MPU6050 wind detection, decision engine, and PID motor movement logic. **Never touches the Wi-Fi API.**
  * **Provisioning:** On first boot or failed connection, spins up `Soltra-Setup` captive portal AP for credential entry without reflashing.

* **The Tactical Nodes (Corners 1, 2, & 3):** `Seeed XIAO ESP32-C3`
  * **Role:** Ultra-low-power peripheral vision devices.
  * **Processing:** Runs localized 1D Time-Series TinyML to mathematically filter out artificial light sources (detecting the 50Hz AC flicker of streetlights vs. DC solar radiation).
  * **Radio:** Sends sensor packets to Hub via ESP-NOW on Hub-commanded polling cycle.

* **The Overwatch Node (Corner 4):** `Seeed XIAO ESP32-S3 Sense`
  * **Role:** Visual command center and primary centroid tracker.
  * **Processing:** Runs a Convolutional Neural Network (Computer Vision) to track the sun's centroid and map incoming cloud cover.
  * **Power Mode:** Camera remains powered off. UV sensor runs 24/7. UV spike → wakes camera for CV tracking.

## 2. Sensor Payload (The Senses)

* **Hub Survival & Analytics Sensors:**
  * **MPU6050:** Measures physical shake and wind vibrations. Accel threshold `>18000` triggers emergency flat-stow. *(Wired: SDA=GPIO1, SCL=GPIO2)*
  * **DS1302 RTC:** Provides exact date/time for the Ephemeris algorithm. *(Wired: DAT=GPIO5, CLK=GPIO6, RST=GPIO4)*
  * **HW-611:** Barometric pressure for local ambient weather tracking.
  * **INA219 / INA226:** Power Profiler measuring actual Wattage generated by the main panel to validate tracking efficiency.

* **Peripheral Sensor Fusion (All 4 Corners):**
  * **KY-018 LDR:** Baseline visible brightness tracking.
  * **GUVA-S12SD UV Sensor:** Pure solar radiation detection (artificial light rejection).
  * **TSL2591 Spectrometer:** Dual-band Infrared vs. Visible light ratio calculation.

* **Hardware Tripwire (Corner 4 Specific):**
  * The UV sensor runs 24/7 at ultra-low power. A massive UV spike interrupts the S3 Sense to boot the camera and AI inference pipeline.

---

## 3. Power Grid (The Energy)

* **Distributed Edge Power:** Every remote node (Corners 1-4) operates entirely off-grid using its own **5000mAh LiPo Battery**, trickle-charged by a **5W Type-C Solar Panel**.
* **Integrated BMS:** Seeed XIAO architecture handles LiPo voltage regulation and deep-sleep protection natively in silicon.
* **The Pager Protocol:** Corner 4 utilizes a 10-second micro-nap cycle. Radio wakes for ~0.1s to check for commands → average draw drops to ~1.6mA → months of battery life while keeping the camera ready within 10 seconds for "Showcase Mode".

---

## 4. Telemetry & Network (The Nervous System)

* **Zero-Collision ESP-NOW Polling:** The Hub acts as radio master. It individually pings each corner in a sequenced loop, eliminating packet collisions.
* **Cloud Broker: HiveMQ Serverless** *(Phase 5 — LIVE)*
  * **Protocol:** MQTT over TLS — Heltec uses port `8883`, web dashboard uses WSS port `8884`.
  * **Topic — Telemetry (Hub → Cloud → Dashboard):** `helios/telemetry`
  * **Topic — Manual Control (Dashboard → Cloud → Hub → Motor Node):** `helios/control/manual`
  * **Topic — Heartbeat:** `helios/status`
  * **Publish Interval:** Every 5 seconds.
  * **Payload (Live):** `{"wind_speed":12.4,"solar_yield":847.3,"panel_angle":184.2,"wind_alert":false,"light_level":0,"node_online":false,"status":"sensor_offline"}`
* **Predictive API Routing:** Hub connects to weather service APIs (Kuala Lumpur). Approaching storm cell → panel stows *before* the physical wind sensor feels a breeze.

---

## 5. Software Intelligence (The Logic)

* **Ephemeris Algorithm:** Calculates exact minute of local sunrise/sunset using orbital math and the DS1302 RTC, enforcing a mathematical night-time mechanical lock overriding all sensors.
* **Decision Engine:** Runs every 1 second on Core 1. Priority order: (1) Wind Alert → Emergency Stow, (2) Night Hours → Reset East & Sleep, (3) Sensor Node Offline → Alert, (4) Nominal → Tracking.
* **PID Control Loop:** Calculus-based motor control (Proportional-Integral-Derivative) ensuring smooth actuator acceleration/deceleration. *(Integration pending)*

## 6. Operator HUD & AI Interface

* **Technology Stack:** React (Vite) + React Router + Three.js (`@react-three/fiber`) + Framer Motion + Lucide React
* **Aesthetic:** High-Craft Industrial Terminal. Replaces legacy glassmorphism with stark data-dense styling — `Bebas Neue` & `monospace` fonts, CRT scanline overlays, skewed panels, mechanical crosshairs, dark `#001a24` backgrounds, `#00d9ff` primary cyan, and `#ff2a2a` alert red accents.
* **Frontend Architecture (`software/soltra-hud/`):**
  * **Main Routing:** `/` (Main Menu), `/hub` (Sub-menu), `/system` (AI Overseer), `/logs` (Hardware Logs), `/network` (Network Grid).
  * **`SystemHub.jsx` (AI Overseer):** Features the interactive 3D VRM Avatar (`VrmAvatar.jsx`) with multi-animation cross-fading, auto-blink, and facial expression overrides. Includes telemetry readouts (Wind, Irradiance, Azimuth) and an LLM command terminal.
  * **`ResumePage.jsx` (Hardware Logs):** Diagnostic UI for actuator hardware arrays with static system logs and drill-down views.
  * **`Socials.jsx` (Network Grid):** Infrastructure health monitor displaying Node status, connection latency, ping packet loss, and sync logs.
* **Standalone Web Dashboard (`software/soltra-dashboard/`):** Legacy single-file industrial HUD that connects directly to HiveMQ via WebSocket (`mqtt.js` CDN). Displays live telemetry and manual D-pad override buttons.

## 7. Voicebox — Local AI Voice Backend

* **Location:** `software/soltra-tts/`
* **Runtime:** Python 3.10, CUDA 12.1 (GPU-accelerated)
* **Role:** Provides high-fidelity, low-latency text-to-speech for the HUD avatar's spoken responses.
* **TTS Pipeline:** Primary — `chatterbox-tts` / `Qwen3-TTS`. Fallback — browser Web Speech API.
* **LLM Integration:** `llmService.js` sends operator commands to the backend, which returns a structured response `{text, expression}` to drive both the chat display and avatar facial state simultaneously.
* **Setup:** `tools/setup_voicebox.ps1` / `tools/run_voicebox_setup.ps1` automate environment creation with PyTorch CUDA and all dependencies.
